# OTel pipeline fix — 2026-05-21

## Symptom

Traces and metrics had stopped showing up in the dashboard. Cause unknown —
could have been the local Docker collector, the Azure VM collector, or auth.

## Root cause

**The OTel JS SDK that ships inside Claude Code emits Sum (counter) metrics
with DELTA aggregation temporality by default. The `prometheusremotewrite`
exporter only supports CUMULATIVE — it silently drops anything else.**

The only Claude data that ever reached Prometheus was the synthesized
`target_info` gauge, which is generated by the exporter from resource
attributes and bypasses the temporality check. Every actual data point
(`claude_code.token.usage`, `claude_code.cost.usage`,
`claude_code.active_time.total`, etc.) was discarded silently — no error,
no warning, no metric counter visible from the outside.

Copilot CLI was unaffected because the Go SDK emits cumulative by default.

The bug had been latent since whenever Claude Code's enhanced telemetry was
first enabled. It looked like "Copilot works, Claude doesn't" — which is
exactly what you'd see if Claude's metrics were being thrown away upstream
of Prom and nobody noticed.

A contributing factor: Docker Desktop was not running when the investigation
started, so the local OTel collector container was down, and the user assumed
the whole pipeline was broken. After Docker was restarted, the LOCAL side
came back up and (we later confirmed) was correctly forwarding every metric
batch to the server. The actual production bug — silent delta-drop on the
server — would have been there even with Docker up.

## How I found it

Methodical layer-by-layer probe of the pipeline:

1. User env vars set correctly. Docker Desktop was off → restarted. Container
   `restart: unless-stopped` brought the local collector back online.
2. Verified the local collector accepts OTLP POSTs (HTTP 200) and that the
   Azure VM is reachable on :4318 (TCP open, OIDC returns 401 to unauth
   requests — correct).
3. Enabled `debug` exporter with `verbosity: normal` on the local collector.
   Confirmed Claude emits all expected metrics every 60 s:
   `lines_of_code.count`, `cost.usage`, `token.usage`, `code_edit_tool.decision`,
   `active_time.total`.
4. SSH'd into the VM (key pulled from Key Vault), enabled verbose logging
   on the server collector. Confirmed every metric arrives at the server
   with correct data points and labels.
5. Queried Prom directly: only `target_info{job="claude-code"}` had current
   samples. The 20 metric names in Prom's index were stale entries from old
   Copilot data — none had current samples either.
6. Tried `add_metric_suffixes: false` — no change. Tried pinning to collector
   v0.139.0 (matches local working version) — no change. Both ruled out as
   the cause.
7. Added a `file` exporter to the server's metrics pipeline. Dumped one
   batch to disk. Found `"aggregationTemporality":1` (DELTA) on every sum.
   Prometheus only stores cumulative. Mystery solved.

## What I changed

### Server-side (`azure-deploy/server/otel-collector-config.yaml`)

Added the `deltatocumulative` processor to the metrics pipeline. It buffers
deltas and emits running cumulative totals, which Prom can store.

```diff
 processors:
   batch:
+  deltatocumulative:

 service:
   pipelines:
     metrics:
       receivers: [otlp]
-      processors: [batch]
+      processors: [deltatocumulative, batch]
       exporters: [prometheusremotewrite, debug]
```

Applied the same change directly on the VM (`~/otel-stack/otel-collector-config.yaml`)
and restarted the container.

### Client-side (`azure-deploy/client/setup-machine.ps1`)

Added `OTEL_EXPORTER_OTLP_METRICS_TEMPORALITY_PREFERENCE=cumulative` as a
persistent user env var (both `[Environment]::SetEnvironmentVariable` and
`$env:`). Defense in depth: even if the server-side processor gets removed,
clients will send cumulative directly so Prom stores them natively.

Also set the var on this machine immediately. Already-running Claude/Copilot
processes inherit env from their parent shell; new shells (and the next
Claude/Copilot launch) will pick up the var.

### Local collector (`azure-deploy/client/otel-collector-config.yaml`)

Reverted the temporary diagnostic changes (debug exporter wired into all
pipelines, verbosity bumped to normal). Back to clean steady state.

## How to keep this from breaking again

- The Azure VM's collector config now has the processor in place. Anyone
  redeploying from `azure-deploy/server/` will get the fix automatically.
- New client machines run `setup-machine.ps1`, which sets the cumulative
  preference env var. Even if the server processor is ever removed, new
  clients are unaffected.
- Existing client machines that ran the old `setup-machine.ps1`: re-run it
  to add the temporality env var. Without that, those clients still depend
  on the server-side processor — but the processor is enough.
- The `otel/opentelemetry-collector-contrib:latest` tag on the server may
  still float and pull breaking changes silently. Recommend pinning to
  `:0.139.0` (already done on the VM, but `:latest` is still in
  `azure-deploy/server/docker-compose.yaml` — see "Known follow-ups").
